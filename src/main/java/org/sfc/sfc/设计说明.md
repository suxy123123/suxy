1. 用户管理模块：登录、注册、角色权限分配

## 四、数据库设计（完整建表脚本在resource包下）
### 1. 用户表（t_sys_user）
- id：BIGINT 主键，自增
- username：VARCHAR(50) 登录账号，唯一
- password：VARCHAR(100) 密码（MD5加密）
- real_name：VARCHAR(50) 真实姓名
- role_code：VARCHAR(50) 角色编码（ADMIN/EDITOR/REVIEWER）
- status：VARCHAR(20) 状态（NORMAL/DISABLED）
- create_time：DATETIME 创建时间
- update_time：DATETIME 更新时间

### 2. 角色表（t_sys_role）
- id：BIGINT
- role_code：VARCHAR(50)
- role_name：VARCHAR(50)

### 3. 稿件表（t_manuscripts）
- id：BIGINT 主键
- title：VARCHAR(255) 稿件标题
- content：TEXT 稿件内容
- status：VARCHAR(50) 状态（DRAFT/PENDING_1/PENDING_2/PENDING_3/REJECTED/COMPLETED）
- reject_reason：TEXT 退回原因
- last_review_stage：VARCHAR(50) 上一审核阶段
- create_user：BIGINT 创建人ID
- create_time：DATETIME
- update_time：DATETIME

### 4. 审核记录表（t_review_records）
- id：BIGINT 主键
- manuscript_id：BIGINT 稿件ID
- operation：VARCHAR(50) 操作类型（SUBMIT/APPROVE/REJECT/RESUBMIT）
- from_status：VARCHAR(50) 原状态
- to_status：VARCHAR(50) 目标状态
- comment：TEXT 审核意见
- operate_user：BIGINT 操作人ID
- operate_time：DATETIME



## 五、枚举设计

### 2. 操作类型枚举（OperationTypeEnum）

- APPROVE("APPROVE", "通过")
- REJECT("REJECT", "退回")
- RESUBMIT("RESUBMIT", "重提")

## 六、业务流程
1. 撰稿人创建稿件 → 状态：DRAFT（草稿）
2. 撰稿人提交审核 → 状态：PENDING_1（待一审）
3. 一审通过 → PENDING_2（待二审）
4. 二审通过 → PENDING_3（待三审）
5. 三审通过 → COMPLETED（已完成）
6. 任意环节退回 → REJECTED（已退回），记录原环节
7. 撰稿人重提 → 恢复至上一审核环节
8. 实现逻辑：考虑到项目比较简单，没有引入流程引擎，只用了一张表记录每个稿件的流程节点。对某个稿件，每有一次流程操作，就往流程记录表入库一条记录，记录本次流程的状态、操作时间、上次状态等信息，通过manuscript_id字段跟稿件表关联。

## 七、核心API接口
- POST /account/login：用户登录
- POST /account/register：用户注册


- POST /manuscripts/createManuscripts：创建稿件
- POST /manuscripts/editManuscripts：编辑稿件
- GET /manuscripts/pageList：稿件列表分页查询
- GET /manuscripts/getDetailById：稿件详情

- POST /manuscripts/{id}/submitPending1：提交一审
- POST /manuscripts/{id}/approve：审核通过
- POST /manuscripts/{id}/reject：审核退回
- POST /manuscripts/{id}/resubmit：重新提交
## 八、权限控制实现
- 采用注解+拦截器模式：@RequireRole("EDITOR")
- 登录拦截：未登录返回401
- 权限拦截：无权限返回403
- 如要关闭，将WebConfig类的注解@Configuration去掉，整个项目将不再进行用户校验
