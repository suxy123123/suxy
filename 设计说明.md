## 一、前后端依赖以及启动
1. maven仓库使用的是阿里云国内镜像
2. 其他依赖和版本依赖关系在项目pom文件中已处理，直接将项目包引入idea即可启动
3. 后端项目jdk使用jdk17，前端使用node.js版本是V20.20.2
4. 启动后端
5. 安装好node.js后在前端项目/frontend目录下运行 `npm install` 安装依赖包，再运行 `npm run dev` 启动项目
6. 项目运行成功后，访问 `http://localhost:3000/` 即可进入项目首页

## 二、功能模块
1. 用户管理模块：登录、注册、角色权限分配
2. 权限控制模块：基于角色的接口与数据权限
3. 稿件管理模块：创建、编辑、提交、查询稿件
4. 三级审核模块：草稿、一审、二审、三审逐级流转
5. 审核记录模块：操作日志、审核意见、历史追溯

## 三、角色与权限

系统采用**基于数据库的动态权限控制（RBAC）**，权限配置存储在 `t_sys_role_permission` 表中，支持灵活调整。

### 权限标识说明
| 权限标识 | 说明 | 使用场景 |
|---------|------|----------|
| `manuscript:view` | 查看稿件 | 所有角色 |
| `manuscript:create` | 创建稿件 | 作者、管理员 |
| `manuscript:edit` | 编辑稿件 | 作者、管理员 |
| `manuscript:submit` | 提交审核 | 作者、管理员 |
| `manuscript:approve` | 审核通过 | 审核员、管理员 |
| `manuscript:reject` | 审核退回 | 审核员、管理员 |
| `manuscript:resubmit` | 重新提交 | 作者、管理员 |

### 角色权限对照表

#### 1. 管理员（ADMIN）
- **权限范围**：拥有全部 7 项权限
- **能力**：可以操作任何稿件的任何流程（创建、编辑、提交、审核、退回、重提）
- **适用场景**：系统管理员，处理特殊情况或干预流程

#### 2. 作者（EDITOR）
- **权限范围**：5 项权限（view、create、edit、submit、resubmit）
- **能力**：
  - ✅ 创建稿件
  - ✅ 编辑草稿或已退回的稿件
  - ✅ 提交稿件进入审核流程
  - ✅ 被退回后重新提交
  - ❌ 无审核权限（不能通过/退回）

#### 3. 一审员（PENDING_1）
- **权限范围**：3 项权限（view、approve、reject）
- **能力**：
  - ✅ 查看稿件
  - ✅ 审核通过（稿件状态为 PENDING_1 时）
  - ✅ 审核退回（稿件状态为 PENDING_1 时）
  - ❌ 无创建/编辑/提交权限

#### 4. 二审员（PENDING_2）
- **权限范围**：3 项权限（view、approve、reject）
- **能力**：
  - ✅ 查看稿件
  - ✅ 审核通过（稿件状态为 PENDING_1 或 PENDING_2 时）
  - ✅ 审核退回（稿件状态为 PENDING_1 或 PENDING_2 时）
  - ❌ 无创建/编辑/提交权限
  - 📌 说明：二审员权限高于一审员，可越级审核

#### 5. 三审员（PENDING_3）
- **权限范围**：3 项权限（view、approve、reject）
- **能力**：
  - ✅ 查看稿件
  - ✅ 审核通过（稿件状态为 PENDING_1、PENDING_2 或 PENDING_3 时）
  - ✅ 审核退回（稿件状态为 PENDING_1、PENDING_2 或 PENDING_3 时）
  - ❌ 无创建/编辑/提交权限
  - 📌 说明：三审员权限最高，可越级审核所有环节

### 权限控制实现
- **前端**：通过角色编码（roleCode）与稿件状态（status）匹配判断按钮显示
  - 一审员只能操作 PENDING_1 状态
  - 二审员可操作 PENDING_1、PENDING_2 状态
  - 三审员可操作 PENDING_1、PENDING_2、PENDING_3 状态
- **后端**：登录时查询 `t_sys_role_permission` 表，返回权限列表；Service层校验稿件当前状态与角色权限匹配
- **灵活性**：修改权限只需调整数据库配置，无需改动代码

### 权限层级关系
```
管理员（ADMIN）> 三审员（PENDING_3）> 二审员（PENDING_2）> 一审员（PENDING_1）
```
- 上级角色可以操作下级角色对应状态的稿件
- 同一级别角色只能操作对应状态的稿件

## 四、数据库设计（完整建表脚本在resource/sql包下）

### 1. 用户表（t_sys_user）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| username | VARCHAR(50) | 登录账号，唯一 |
| password | VARCHAR(100) | 密码（MD5加密） |
| real_name | VARCHAR(50) | 真实姓名 |
| role_code | VARCHAR(50) | 角色编码（ADMIN/EDITOR/PENDING_1/PENDING_2/PENDING_3） |
| status | VARCHAR(20) | 状态（NORMAL/DISABLED） |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 2. 权限表（t_sys_permission）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| permission_name | VARCHAR(50) | 权限名称（如"查看稿件"） |
| permission_key | VARCHAR(50) | 权限标识（如"manuscript:view"），唯一 |

### 3. 角色权限关联表（t_sys_role_permission）
| 字段 | 类型 | 说明 |
|------|------|------|
| role_code | VARCHAR(50) | 角色编码 |
| permission_key | VARCHAR(50) | 权限标识 |
| 联合主键 | (role_code, permission_key) | 保证一个角色的权限不重复 |

### 4. 稿件表（t_manuscripts）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| title | VARCHAR(255) | 稿件标题 |
| content | TEXT | 稿件正文内容 |
| status | VARCHAR(50) | 当前状态（DRAFT/PENDING_1/PENDING_2/PENDING_3/REJECTED/COMPLETED） |
| reject_reason | TEXT | 审核退回原因（仅REJECTED状态有值） |
| last_review_stage | VARCHAR(50) | 上一审核阶段（用于退回后重提） |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 5. 审核记录表（t_review_records）
| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，自增 |
| manuscript_id | BIGINT | 稿件ID（外键关联） |
| operation | VARCHAR(50) | 操作类型（CREATE/SUBMIT/APPROVE/REJECT/RESUBMIT） |
| from_status | VARCHAR(50) | 原状态 |
| to_status | VARCHAR(50) | 目标状态 |
| comment | TEXT | 审核意见/备注 |
| operate_time | DATETIME | 操作时间 |
| 索引 | idx_manuscript_id | 按稿件ID查询优化 |



## 五、业务流程
1. 撰稿人创建稿件 → 状态：DRAFT（草稿）
2. 撰稿人提交审核 → 状态：PENDING_1（待一审）
3. 一审通过 → PENDING_2（待二审）
4. 二审通过 → PENDING_3（待三审）
5. 三审通过 → COMPLETED（已完成）
6. 任意环节退回 → REJECTED（已退回），记录原环节
7. 撰稿人重提 → 恢复至上一审核环节
8. 实现逻辑：考虑到项目比较简单，没有引入流程引擎，只用了一张表记录每个稿件的流程节点。对某个稿件，每有一次流程操作，就往流程记录表入库一条记录，记录本次流程的状态、操作时间、上次状态等信息，通过manuscript_id字段跟稿件表关联。

### 审核权限说明
- **越级审核**：二审员和三审员具备越级审核权限
  - 二审员可审核 PENDING_1 和 PENDING_2 状态的稿件
  - 三审员可审核 PENDING_1、PENDING_2 和 PENDING_3 状态的稿件
- **权限隔离**：一审员只能审核 PENDING_1 状态的稿件
- **退回机制**：退回后稿件状态变为 REJECTED，记录退回原因和上一审核阶段，作者修改后可重新提交

## 六、核心API接口
- POST /account/login：用户登录
- POST /account/register：用户注册


- POST /manuscripts/createManuscripts：创建稿件
- POST /manuscripts/editManuscripts：编辑稿件
- GET /manuscripts/pageList：稿件列表分页查询
- GET /manuscripts/getDetailById：稿件详情

- POST /manuscripts/{id}/submitPending1：提交一审
- POST /manuscripts/{id}/approve：审核通过
- POST /manuscripts/{id}/reject：审核退回
- POST /manuscripts/{id}/resubmit：重新提交
